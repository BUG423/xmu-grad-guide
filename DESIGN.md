## 一、项目目标

将厦门大学信息学院研究生毕业流程指引从一篇"长篇文字堆砌"的网页，改造成一个 **交互式、分阶段、可追踪** 的毕业流程指引 Web 应用。

### 核心功能

1. **身份选择**：学生登录后选择自己的学位类型，系统展示对应流程
2. **阶段式指引**：将毕业流程拆分为若干阶段，每个阶段包含具体任务清单
3. **勾选追踪**：学生可勾选已完成的任务，系统记录进度
4. **进度可视化**：Dashboard 展示整体完成度，每个阶段显示进度条
5. **材料清单**：每个任务附带所需材料、注意事项、相关链接

## 二、学位类型分类

根据原始页面内容，本流程指引覆盖 **3 种学位类型**。工程硕博（卓越工程师计划等）由学院另行通知，不在本流程范围内。

| 序号 | 身份标识 | 名称 | 关键特征 |
|------|---------|------|---------|
| 1 | `academic-master` | 学术型硕士 | 无预答辩、2份评阅、4本论文 |
| 2 | `professional-master` | 专业型硕士 | 企业专家评阅、实践成果可选、实习实践报告 |
| 3 | `academic-phd` | 学术型博士 | 必须预答辩、3份评阅、5本论文、质量核查、答辩海报 |

> ⚠️ **工程硕博**（工程硕士/工程博士，含卓越工程师计划等）：由学院另行通知，不适用本流程。选择此身份时，页面将提示联系相关负责老师获取单独通知。

### 各类型差异总览

| 流程节点 | 学术硕士 | 专业硕士 | 学术博士 |
|---------|:-------:|:-------:|:-------:|
| 预答辩 | ❌ 导师决定 | ❌ 导师决定 | ✅ 必须 |
| 实践成果鉴定会 | ❌ | 可选 | ❌ |
| 评阅人数 | 2人 | 2人(含1企业) | 3人 |
| 论文质量核查 | ❌ | ❌ | ✅ 必须 |
| 答辩海报PPT | ❌ | ❌ | ✅ 必须 |
| 论文字数下限 | 3万字 | 3万字 | 5万字 |
| 提交论文份数 | 4本 | 4本 | 5本 |
| 实习实践报告 | ❌ | ✅ | ❌ |
| 光盘额外内容 | 基础 | 基础 | +PPT+海报+摘要 |

## 三、技术方案

### 3.1 推荐方案：React 全栈应用

| 层次 | 技术选型 | 理由 |
|------|---------|------|
| **前端框架** | React 18 + TypeScript | 生态丰富，UI 组件库多 |
| **UI 组件库** | Ant Design 5 | 中文友好，Steps/Checkbox/Progress 组件完善 |
| **路由** | React Router v6 | SPA 页面切换 |
| **状态管理** | Zustand | 轻量，适合中等复杂度 |
| **后端框架** | Express.js (Node.js) | 简单直接，适合校园内网部署 |
| **数据库** | SQLite (开发) / PostgreSQL (生产) | SQLite 免安装，方便快速启动 |
| **ORM** | Prisma | 类型安全，迁移方便 |
| **认证** | JWT + 本地账号 或 CAS 统一认证 | 灵活适配校园环境 |
| **部署** | Docker / 直接 Node.js 运行 | 校园服务器友好 |

### 3.2 替代方案

| 方案 | 适合场景 |
|------|---------|
| **纯前端 + localStorage** | 无需用户系统，个人使用，最快上线 |
| **Next.js 全栈** | 需要 SSR/SEO，更现代的技术栈 |
| **Vue 3 + Element Plus** | 团队熟悉 Vue 生态 |

**建议先从纯前端方案开始**，快速验证体验；后续按需加入后端。

## 五、数据模型设计

### 5.1 用户表 `User`

```
id          UUID        PK
username    String      Unique    学号/工号
password    String      Hashed
name        String                姓名
email       String                邮箱
degreeType  Enum        学位类型
createdAt   DateTime
updatedAt   DateTime
```

### 5.2 流程阶段表 `Phase`

```
id          UUID        PK
degreeType  Enum        所属学位类型
order       Int         阶段序号 (1,2,3)
title       String      阶段名称（如"定稿送审"）
description String      阶段描述
```

### 5.3 任务表 `Task`

```
id          UUID        PK
phaseId     UUID        FK → Phase
order       Int         任务序号
title       String      任务名称
description Text        任务详细描述
materials   JSON        所需材料列表 [{name, required, format}]
links       JSON        相关链接 [{title, url}]
warnings    Text        注意事项
deadline    String      参考截止时间（如"3月底"）
```

### 5.4 用户进度表 `UserProgress`

```
id          UUID        PK
userId      UUID        FK → User
taskId      UUID        FK → Task
completed   Boolean     是否完成
completedAt DateTime    完成时间
notes       Text        用户备注
```

## 六、流程阶段 & 任务清单（按学位类型）

### 6.1 学术型硕士 (`academic-master`)

#### 阶段一：定稿送审（8个任务）

| 序号 | 任务 | 关键材料/操作 |
|------|------|------------|
| 1.1 | 毕业图像采集 | 按通知完成采集，用于毕业证/学位证/学信网 |
| 1.2 | 科研成果审核 | 检索证明+论文全文+导师签字扫描 |
| 1.3 | 学分及前置学历审核 | 系统检查学分+前置学历学位 |
| 1.4 | 论文定稿 | 导师同意定稿，≥3万字，按学校格式排版 |
| 1.5 | 系统送审申请 | 本研系统申请+导师审核+导出《送审申请表》 |
| 1.6 | 第一次查重 | PDF全文(不含声明)、<20M、邮件发秘书(抄送导师) |
| 1.7 | 论文形式审查 | 电子版审查论文格式 |
| 1.8 | 论文送审（盲审） | 盲审PDF+信息汇总表+摘要TXT，教育部平台，≥2评阅人 |

#### 阶段二：答辩（4个任务）

| 序号 | 任务 | 关键材料/操作 |
|------|------|------------|
| 2.1 | 确认科研成果审核 | 再次确认 |
| 2.2 | 确定答辩委员会和秘书 | 导师提交秘书名单给研究生秘书 |
| 2.3 | 准备答辩论文 | 打印若干份，非最终版；差错率<万分之二 |
| 2.4 | 进行答辩 | 秘书记录和录音；答辩后修改论文 |

#### 阶段三：答辩后提交材料（8个任务）

| 序号 | 任务 | 关键材料/操作 |
|------|------|------------|
| 3.1 | 论文修改定稿 | 导师同意，更新系统论文信息 |
| 3.2 | 第二次查重 | 全文PDF，<20M，附《答辩决议》扫描件 |
| 3.3 | 系统学位申请 | 打印《学位申请表》2份，彩色签字 |
| 3.4 | 提交纸质材料 | 按档案目录整理，装入档案袋，共20项材料 |
| 3.5 | 论文形式审查 | 先交1本(无须胶装)审查 |
| 3.6 | 胶装论文提交 | 审查通过后交4本胶装论文 |
| 3.7 | 学位论文光盘 | 1份，含PDF+WORD+答辩录音 |
| 3.8 | 图书馆论文提交 | 自行送交图书馆1本 |

### 6.2 专业型硕士 (`professional-master`)

在学术型硕士基础上增加/调整：

| 差异 | 说明 |
|------|------|
| + 实践成果鉴定会 | 以实践成果申请学位者，在送审前完成 |
| + 实习实践报告表 | 答辩后提交材料时额外提交 |
| 评阅专家要求 | 需1名行（企）业专家评审 |
| 送审途径 | 同时走教育部平台+学院系统 |
| 档案目录 | 使用《专业型研究生学籍学位档案卷内目录》 |

### 6.3 学术型博士 (`academic-phd`)

在学术型硕士基础上增加/调整：

| 差异 | 说明 |
|------|------|
| **+ 预答辩** | 定稿后必须完成，在正式答辩3个月前进行 |
| **+ 博士论文质量核查** | 送审结果返回后，答辩前完成 |
| **+ 答辩海报PPT** | 答辩前2个工作日提交秘书 |
| 评阅人数 | ≥3名校外博导/正高级专家 |
| 论文字数 | ≥5万字 |
| 提交论文份数 | 5本胶装 |
| 光盘内容 | +答辩PPT +海报PPT +中英文摘要 |
| 评阅周期 | 35个工作日 |
| 提交额外材料 | 预答辩申请表 + 预答辩成绩评定表 |

#### 学术型博士任务清单（约18个任务）

**阶段一：定稿送审（11个任务）**
1. 毕业图像采集
2. 科研成果审核
3. 学分及前置学历审核
4. 论文初稿定稿（≥5万字）
5. **预答辩申请**（导师同意→系统申请→学院审核）
6. **预答辩**（学院统一组织）
7. 预答辩后修改再次定稿
8. 系统送审申请
9. 第一次查重（预答辩通过后方可）
10. 论文形式审查
11. 论文送审盲审（≥3评阅人，35个工作日）

**阶段二：答辩（6个任务）**
1. **博士论文质量核查**（提交盲审论文纸质+电子版）
2. 根据评阅意见修改论文
3. 确认科研成果审核
4. 确定答辩委员会和秘书
5. **制作答辩海报PPT**（答辩前2天提交）
6. 进行答辩

**阶段三：答辩后提交材料（9个任务）**
1. 论文修改定稿
2. 第二次查重
3. 系统学位申请
4. 提交纸质材料（含预答辩材料）
5. 论文形式审查
6. 胶装论文提交（5本）
7. 学位论文光盘（含PPT+海报+摘要+录音）
8. 图书馆论文提交
9. 预答辩纸质材料提交

### 6.4 工程硕博（另行通知）

工程硕士、工程博士（含卓越工程师计划等特殊项目）的毕业流程由学院**另行单独通知**，不在本流程指引范围内。

选择此身份的学生将看到提示页面，包含：
- 联系负责老师的联系方式
- 建议关注学院通知公告
- 部分通用材料（如图像采集、论文格式规范）仍可参考本系统

## 七、论文撰写小贴士 ✍️

> 本节为额外补充内容，原始页面未详细展开，但根据实际经验整理，帮助学生合理规划论文撰写时间。

### 7.1 时间倒推规划

以 **4月初提交送审论文** 为目标的倒推时间线：

| 时间节点 | 事项 | 说明 |
|---------|------|------|
| **4月初** | 提交送审论文 | 盲审论文 + 查重 + 形式审查 |
| **3月中下旬** | 论文定稿 | 导师确认同意，格式调整完毕 |
| **3月中旬** | 第一次查重 + 送审申请 | 预留查重不通过需修改的时间 |
| **2月下旬-3月** | 论文修改完善 | 导师审阅反馈，多轮修改 |
| **1月-2月** | 完成论文初稿 | 寒假是写论文的关键时期 |
| **前一年10-12月** | 确定论文选题和研究框架 | 尽早与导师沟通确定方向 |
| **整个研二/博三** | 积累研究成果（小论文等） | 科研成果是送审的前提 |

### 7.2 工作量参考

| 学位类型 | 建议工作量（创新点/章节） | 论文字数下限 | 建议准备周期 |
|---------|----------------------|------------|------------|
| 学术型硕士 | **2个**创新点/工作 | ≥3万字 | 6-12个月 |
| 专业型硕士 | **2个**工作（可含实践成果） | ≥3万字 | 6-12个月 |
| 学术型博士 | **3个**创新点/工作 | ≥5万字 | 12-24个月 |

> 💡 "2个工作量"的含义：论文中应有2个相对独立且有创新性的研究点或技术贡献，通常对应2-3个核心章节。硕士的2个工作可以是一个方法改进 + 一个应用验证；博士的3个工作应构成一个有机的研究体系。

### 7.3 论文撰写建议

1. **尽早动笔**：不要等"准备好"再写，有了初步想法就开始写绪论和文献综述
2. **先搭骨架**：先确定章节目录结构，与导师确认后再填充内容
3. **格式先行**：一开始就使用学校模板（Word/LaTeX），不要最后再调格式
4. **图表规范**：图表是论文的"门面"，要清晰美观，中文图表标题规范
5. **参考文献管理**：使用 Zotero/EndNote 管理文献，避免最后手忙脚乱
6. **定期与导师沟通**：建议每2-4周给导师看一次进展，避免方向跑偏
7. **留足修改时间**：初稿完成后至少留2-4周用于反复修改打磨
8. **查重意识**：引用规范、用自己的话写，避免大段复制粘贴

## 八、关键设计决策

### 8.1 为什么先做纯前端版？

1. **快速验证**：1-2天即可出可用的原型，让同学试用
2. **无需服务器**：可直接部署到 GitHub Pages / Vercel
3. **数据在本地**：学生的勾选进度存 localStorage，无需担心隐私
4. **渐进增强**：后续再加后端，数据结构已设计好，迁移成本低

### 8.2 流程数据的可配置化

所有流程数据以 JSON/TS 对象形式定义，便于后续：
- 学院政策变化时快速更新
- 管理员通过后台调整
- 支持多学院复用

### 8.3 设计原则

- **移动优先**：大部分学生用手机查看
- **信息分层**：概览简单，点击展开详情
- **降低焦虑**：清晰的进度条，已完成事项一目了然
- **链接直达**：所有系统链接、文档下载直达，减少跳转查找
